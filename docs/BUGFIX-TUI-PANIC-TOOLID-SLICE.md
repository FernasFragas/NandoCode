# TUI Panic: `slice bounds out of range [:8] with length 6`

Status: **Completed (2026-05-09)** — fixes implemented and regression tests added.
Severity: **Critical** — crashes the REPL/TUI mid-session, killing in-progress agent runs.
Owner: TUI / agent integration.

---

## 1. Symptom

Running `nandocodego` and asking the agent to analyse a file or the project crashes the TUI with:

```
runtime error: slice bounds out of range [:8] with length 6
...
github.com/FernasFragas/nandocodego/internal/tui.(*Model).renderToolPanel
        internal/tui/app.go:1132
github.com/FernasFragas/nandocodego/internal/tui.(*Model).renderTranscript
        internal/tui/app.go:1010
github.com/FernasFragas/nandocodego/internal/tui.(*Model).refreshViewportContent
        internal/tui/app.go:972
github.com/FernasFragas/nandocodego/internal/tui.(*Model).handleAgentEvent
        internal/tui/app.go:688
```

Bubble Tea catches the panic, restores the terminal, and exits with `REPL failed: program was killed: program experienced a panic`.

---

## 2. Root cause

`internal/tui/app.go:1132` (in `renderToolPanel`) unconditionally slices the tool ID:

```go
content := fmt.Sprintf("🔧 %s (%s): %s", item.ToolName, item.ToolID[:8], item.Content)
```

Tool IDs are generated by the agent as `fmt.Sprintf("tool-%d", idx)` in:

- `internal/agent/tools.go:36`
- `internal/agent/agent.go:297`, `agent.go:400`
- `internal/agent/speculative.go:123`, `153`, `191`

For any single-digit index (`tool-0` … `tool-9`) the resulting ID is **6 bytes**, so `id[:8]` panics. The first tool call of every run hits this — that is exactly the path that "analyse a file/project" takes (it dispatches a `read_file` / `list_dir` style tool on turn one).

**A second site has the identical bug:** `internal/tui/app.go:596`, inside the `agent.ToolUseProgress` case, also does `e.ID[:8]`. It hasn't surfaced in this stack trace because rendering happens before that branch fires for tools that don't emit progress, but it will panic in any flow that emits a `ToolUseProgress` event with a short ID.

### Why `[:8]` was chosen
The code is mimicking a UUID-prefix style display (`uuid[:8]`). The author appears to have assumed tool IDs are long opaque strings. They are not — they are short, dense, and predictably under 8 chars for the first ten tool calls.

---

## 3. Reproduction

1. `go run ./cmd/nandocodego` (or `./bin/nandocodego`).
2. Ask any prompt that triggers a tool call (e.g. "read README.md", "list the project").
3. The agent emits `ToolUseStart{ID:"tool-0", Name:"read_file", ...}`.
4. `handleAgentEvent` appends a `TranscriptTool` item, then calls `refreshViewportContent` → `renderTranscript` → `renderToolPanel`, which slices `"tool-0"[:8]` and panics.

---

## 4. Fix

Replace fixed-length slicing with a safe truncation helper. Apply at **both** sites in `internal/tui/app.go`.

### 4.1 Add a helper (top of `internal/tui/app.go`, near other small utilities)

```go
// shortID returns up to n bytes of id, used for compact display in the TUI.
// Safe for any input length (including empty strings).
func shortID(id string, n int) string {
    if len(id) <= n {
        return id
    }
    return id[:n]
}
```

If multi-byte runes ever land in IDs (they won't today, but it's cheap insurance), prefer rune-based truncation:

```go
func shortID(id string, n int) string {
    r := []rune(id)
    if len(r) <= n {
        return id
    }
    return string(r[:n])
}
```

### 4.2 Patch the two call sites

`internal/tui/app.go:596`
```go
// before
m.transcript[i].Content = fmt.Sprintf("[%s] %s", e.ID[:8], dataStr)
// after
m.transcript[i].Content = fmt.Sprintf("[%s] %s", shortID(e.ID, 8), dataStr)
```

`internal/tui/app.go:1132`
```go
// before
content := fmt.Sprintf("🔧 %s (%s): %s", item.ToolName, item.ToolID[:8], item.Content)
// after
content := fmt.Sprintf("🔧 %s (%s): %s", item.ToolName, shortID(item.ToolID, 8), item.Content)
```

### 4.3 Regression test

Add a focused unit test under `internal/tui/` that constructs a `TranscriptItem` with a short ID and asserts `renderToolPanel` does not panic and contains the full short ID:

```go
func TestRenderToolPanel_ShortToolID(t *testing.T) {
    m := newTestModel(t) // existing helper or minimal Model
    item := TranscriptItem{
        Kind:     TranscriptTool,
        ToolName: "read_file",
        ToolID:   "tool-0",
        Content:  "ok",
    }
    out := m.renderToolPanel(item) // must not panic
    if !strings.Contains(out, "tool-0") {
        t.Fatalf("expected short id in output, got %q", out)
    }
}
```

A second test should cover `handleAgentEvent` receiving `ToolUseProgress{ID:"tool-1", ...}` to lock in the fix at line 596.

## 5. Adjacent issues found while investigating

These are not the cause of the reported panic but are smell/risk worth fixing in the same patch series.

### 5.1 Duplicate, fragile `for i := len(m.transcript)-1; i >= 0; i--` lookups
In `handleAgentEvent` (lines ~593–635) the `ToolUseProgress` and both branches of `ToolUseResult` walk `m.transcript` from the tail looking for a matching `ToolID`. Three near-identical loops. Risks:
- O(n) per event; on long transcripts with many tool calls this is wasteful.
- Easy to drift — if one branch is updated (e.g. to also set `Error`), the others can be missed (already visible: only the error branch sets `Error`).
- If the matching `TranscriptTool` item is missing (e.g., result arrives without a prior start due to reordering or a dropped event) the update is silently lost.

**Fix:** maintain a `map[string]int` (toolID → transcript index) on `Model`, populated in `ToolUseStart`, consulted in `ToolUseProgress` / `ToolUseResult`, and pruned on transcript reset/compaction. Centralise the "update tool item" logic into one method `updateToolItem(id string, mut func(*TranscriptItem))`.

### 5.2 `fmt.Sprintf("%v", e.Data)` in `ToolUseProgress`
`e.Data` is `interface{}`; `%v` can produce huge strings (entire stream chunks, file contents) that then end up rendered every refresh. Cap to e.g. 4 KiB and add an ellipsis when truncated. Also avoids accidental ANSI/control-byte injection into the viewport.

### 5.3 No panic guard around per-event rendering
A single bad event currently kills the whole REPL session and discards in-flight conversation state. Wrap `m.refreshViewportContent` (or the body of `handleAgentEvent`) in a `defer recover()` that converts the panic into a `CreateSystemItem("[render error: …]")` transcript entry. This turns a future similar bug into a visible glitch instead of a session-ending crash. Bubble Tea's own recover only fires after the program is already torn down — we want to keep the program alive.

### 5.4 `agent.ToolUseResult` display fallback
```go
resultStr = e.Result.Display
if resultStr == "" && e.Result.Data != nil {
    resultStr = fmt.Sprintf("%v", e.Result.Data)
}
```
Same unbounded `%v` concern as 5.2. Tool results often contain whole file bodies. Truncate before rendering and keep the full payload in `state.ActiveTools` if it's needed elsewhere.

### 5.5 Missing `TranscriptItem.ToolID` invariants
`renderToolPanel` assumes `ToolID` is non-empty. Today `CreateToolItem` always sets it, but nothing enforces this. Add a defensive default in `renderToolPanel` (`if item.ToolID == "" { ... }`) so a future producer that forgets to set it doesn't render `()`.

### 5.6 ID format is not collision-safe across sub-flows
`tool-%d` is reused independently in `agent.go`, `tools.go`, and `speculative.go` (each indexes from its own loop). If two flows ever interleave events into the same transcript (e.g. speculative + canonical), IDs will collide and the index map proposed in 5.1 will alias. Either:
- Prefix per-flow (`canon-tool-%d`, `spec-tool-%d`), or
- Use a monotonically increasing counter on the agent.

Pick one before 5.1 lands; otherwise the map fix introduces a different bug.

### 5.7 Emoji width assumption in `renderToolPanel`
`🔧` is a wide grapheme; some terminals render it as 1 cell, some as 2. Lipgloss panel widths can drift by one column per tool item. Low priority, but if/when we standardise on `uniseg`/`runewidth` for layout, include this.

## 6. Plan for an implementing agent

Execute in this order. Each step is independently testable.

### Step 1 — stop the bleeding (must ship)
1. Add `shortID(id string, n int) string` helper at the top of `internal/tui/app.go` (above `handleAgentEvent`). Use the rune-safe variant from §4.1.
2. Replace `e.ID[:8]` at `app.go:596` with `shortID(e.ID, 8)`.
3. Replace `item.ToolID[:8]` at `app.go:1132` with `shortID(item.ToolID, 8)`.
4. Add unit tests (§4.3) covering: short ID, exactly-8 ID, longer ID, empty ID. Empty ID should render as `()` without panic.
5. `go test ./internal/tui/...` and a manual smoke test (`go run ./cmd/nandocodego` → "read README.md").

**Done when:** the original repro no longer panics and the new tests pass.

### Step 2 — render-loop panic guard
1. In `handleAgentEvent` (or `refreshViewportContent`), add:
   ```go
   defer func() {
       if r := recover(); r != nil {
           m.transcript = append(m.transcript, CreateSystemItem(
               fmt.Sprintf("[render error: %v]", r)))
       }
   }()
   ```
   Place it so the panic message itself is rendered on the next tick (the recover must run before the next refresh, not inside it — easiest is a wrapper method called from `Update`).
2. Add a test that injects a deliberately panicking transcript item (e.g. via a hook) and asserts the program continues.

**Done when:** simulating a bug in any single render path produces a visible `[render error: …]` line instead of killing the session.

### Step 3 — consolidate tool-item lookup
1. Add `toolIndex map[string]int` to `Model`. Initialise in the constructor.
2. Populate it in the `ToolUseStart` case (`m.toolIndex[e.ID] = len(m.transcript)-1` after appending).
3. Replace the three reverse-scan loops at lines ~593, ~619, ~629 with `updateToolItem(id, func(it *TranscriptItem){ ... })`.
4. Clear the map whenever the transcript is reset (find: anywhere `m.transcript = ...` reassigns, including `/clear`-style commands and compaction handlers).
5. Tests: simulate Start → Progress → Result and assert all three updates land on the same item; simulate two interleaved tool calls and assert each updates only its own item.

**Done when:** all three sites use one helper and tests cover interleaving.

### Step 4 — bound rendered tool payloads
1. Introduce `truncateForDisplay(s string, max int) string` (default 4096, append `… (truncated, N more bytes)` suffix).
2. Apply to: the `dataStr` produced in `ToolUseProgress`, and the `resultStr` produced in `ToolUseResult`.
3. Keep the full data in `state.ActiveTools` (already stored as `Summary`) — but truncate `Summary` too if it ever feeds rendering elsewhere. Verify no other reader depends on the full string before clipping there.

**Done when:** rendering a 10MB tool result no longer freezes the TUI.

### Step 5 — namespaced tool IDs (pre-req for any future speculative+canonical merge)
1. Decide on prefix scheme (`canon-`, `spec-`) or central counter. Recommend a central counter on `agent.Agent` to avoid touching every emitter.
2. Update `tools.go`, `agent.go`, `speculative.go` to consume the counter.
3. Update tests that assert on literal `tool-0` strings.

**Done when:** grep for `"tool-%d"` returns one definition.

### Step 6 — small hygiene
- Add `if item.ToolID == ""` guard in `renderToolPanel` (§5.5).
- File a follow-up issue for the emoji-width concern (§5.7); do not block on it.
